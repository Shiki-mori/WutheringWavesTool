# 数据库

*2026/3/24 16:25*
数据分析阶段完成。  
来到这里有两个选择：优化前端显示，或直接入手数据库。  
后者是更好的选择。先做前端将导致后续添加数据库功能时界面管理困难。有了数据库，让数据变“活”，前端才有优化的意义。  

实现路线：  

- 选定技术栈：MySQL  
- 设计最简单表结构，一张核心表  
表结构预览：  

    |字段名|类型|含义|
    |:--:|:--:|:--:|
    |id|bigint|主键|
    |playerId|varchar|用户ID|
    |resourceId|varchar|物品ID|
    |name|varchar|物品名称|
    |isUp|boolean|是否限定|
    |count|int|抽数|  
    |qualityLevel|int|品质
    |poolType|INT|卡池类型|
    |time|datetime|抽卡时间|
    |createdAt|datetime|入库时间|

- 后端改造  
由前端-后端-返回结果的处理流程升级为：  
前端-后端-存数据库-再分析-返回结果  
- 新增接口  
保存数据--POST /gacha/import  
查询数据--GET /gacha/list  
获取统计--GET /gacha/stat  

之后的阶段：  

1. 完善数据库接入，实现存储与查询；
2. 基于数据库重构统计逻辑，实现支持多次导入；
3. 优化前端页面，绘制图表；
4. 完善为完整项目。  

一条实现链：抽卡导入-入库-分析

数据库需要解决的问题：

- 避免重复请求接口（缓存）
- 支持历史数据积累
- 支持复杂统计
- 支持多用户（待扩展）

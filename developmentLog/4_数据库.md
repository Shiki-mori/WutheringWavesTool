## 附录-更改表结构

核心命令：`alter table table_name`。在进行以下所有更改前都需要输入此命令，且末尾不用分号

删除字段（drop column）：`drop column column_name;`

修改字段：  

- 修改字段名称（change）：`change source_name target_name source_type;`
- 修改字段类型（modify）：`modify column_name target_type;`

添加字段（add column）:`add column column_name column_type (after last_column_name);`  
可选择指定位置

修改唯一索引  
本表中唯一索引为：`unique key uniq_record(uid,resource_id,time)`  
删除索引：`drop index uqiq_record on table_name;`  
新增索引：`create unique index uniq_record on table_name (uid,resource_id,time);`

## 通过参数传递，补充saveRecords中填充失败的属性值

gachaIndex和cardPoolId这两个数据缺失。  
cardPoolId从查询参数中传递到saveRecords函数。  
gachaIndex：该参数表示当前数据项是第几抽。有两种方案：先进行统计获取gachaIndex参数，一并存入数据库；或直接从数据库表中删除该属性，后续通过排序方式计算gachaIndex。本项目采用第一种方案，可降低时间复杂度。  
编写一个preprocessing（预处理）js文件，用于获取gachaIndex参数，也可用于处理其他可能需要的参数。最后可以考虑再加一个normalize函数，统一record格式。  

```js
function addGachaIndex(records) {
    if (!Array.isArray(records)) {
        return [];
    }
    const total = records.length;
    return records.map((record, index) => ({
        ...record,//...表示展开record对象的所有属性
        // 官方接口按时间倒序返回，这里补充对应的历史抽数序号。
        gachaIndex: total - index
    }));
}
```

`index`是map自动传入的参数，代表数组当前元素的索引，从0开始。因此`total-index`能正确表示gachaIndex。

## 数据源不提供稳定主键+数据窗口滑动 导致的记录顺序混乱问题

我的设想：更改入库顺序，旧记录先入库，新记录后入库。当新数据入库时，id自动排列到最后，使用id来作为排序标准。  

但是id是数据库技术，不应该用来解决业务问题。业务问题需要使用业务字段解决。id只表示入库顺序，不表示业务顺序。当发生数据库重建、导入迁移、补录旧数据、并发写入这类情形时，id表示顺序完全失效。

去重主键和业务排序应该分开设计。

gacha_index不是数据源稳定主键，不该参与唯一性判断或排序依据。

### 排序层

新增一个本地持久化顺序字段`draw_order`，只在首次入库时赋值，一旦写入，永不更改。表示本地确认的真实抽卡顺序。永久自增业务序号。

将gacha_index修改为：保留time字段，在其末尾添加代表先后顺序的数字；对于time相同的数据，按照抓取返回的顺序排列。由于同一时间的数据总是同时出现，同时过期，这样能避免数据源更新后数据“漂移”的情况。需要重命名为source_order，避免命名混淆。

可以将这两者整合。

ai建议拆字段保存：将代表先后顺序的短字段独立出来，排序时按照time+second_seq的格式。

#### 具体实施

在`preProcessing.js`中新增`in_second_seq`字段。字段值以0~9表示同一time下的顺序。  
当前的入库顺序是最旧的最先入库。当前的抓取结果返回顺序是从最新到最旧排列。那么按照抓取顺序遍历，对于time字段相同的数据项，第一个标记为9......不对。若以后推出五连抽这种形式，这样的表示方法（9,8,7,6,5）就不严谨了。所以先倒序排列，直接计算in_second_seq。
